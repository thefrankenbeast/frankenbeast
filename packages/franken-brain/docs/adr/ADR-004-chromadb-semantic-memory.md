# ADR-004: ChromaDB for Semantic Memory (Vector Store)

- **Date:** 2026-02-19
- **Status:** Accepted
- **Deciders:** franken-brain team

## Context

Semantic memory stores "static but vital" project knowledge — ADR rules, TypeScript coding standards, project-specific documentation — retrieved by semantic similarity. This requires a vector store capable of:

- Embedding text chunks and storing their vector representations
- Cosine/dot-product similarity search returning top-K results
- Metadata filtering (e.g., `project_id = 'X'`)
- Running locally with no paid API keys for development and CI

The project-outline.md explicitly names Pinecone, Milvus, and ChromaDB as options.

## Decision

Use **ChromaDB** (local persistent mode via `chromadb` npm client + local server) as the vector store for Semantic Memory.

- Local server runs via Docker (`chromadb/chroma` image) or Python CLI for development
- In tests: use `chromadb` in-memory ephemeral client mode
- Embeddings generated by a configurable `IEmbeddingProvider` interface (default: `@xenova/transformers` for local, swappable to OpenAI/Anthropic for production)
- Collections namespaced by `project_id` to isolate knowledge bases

**Interface:**
```typescript
interface ISemanticStore {
  upsert(chunks: SemanticChunk[]): Promise<void>;
  search(query: string, topK: number, filter?: MetadataFilter): Promise<SemanticChunk[]>;
  delete(id: string): Promise<void>;
  deleteCollection(projectId: string): Promise<void>;
}
```

## Consequences

### Positive
- Free, open-source, no API key required for local development
- Native metadata filtering reduces post-query filtering logic
- Client library has TypeScript types
- Swappable: `PineconeSemanticStore` and `MilvusSemanticStore` can implement `ISemanticStore` without touching the rest of the module
- In-memory mode for tests means no Docker dependency in CI (test in-memory, validate with Docker in integration tests)

### Negative
- Requires running a Chroma server for integration and production use (Docker dependency)
- Local embedding via `@xenova/transformers` adds ~500MB of model weights to the dev machine
- Not cloud-native; horizontal scaling requires migrating to Pinecone or Milvus

### Risks
- ChromaDB Python server version must stay in sync with the JS client version — pin both in `docker-compose.yml`

## Alternatives Considered

| Option | Pros | Cons | Rejected Because |
|--------|------|------|-----------------|
| Pinecone | Fully managed, scalable | Paid service, requires API key, no local dev mode | Adds cost and external dependency before MVP is validated |
| Milvus | Production-grade, open-source | Complex infra (etcd, MinIO, Milvus server) | Too much operational overhead for local-first MVP |
| `vectra` (local JSON file store) | Zero dependencies, pure TS | No server, no metadata filtering, limited scale | Not a real vector DB; would need replacement anyway |
| pgvector (PostgreSQL extension) | SQL + vector in one DB | Requires Postgres infra; no Node driver with great TS support | Combined with ADR-003 still needs two DB systems |
