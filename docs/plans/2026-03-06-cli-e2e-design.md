# Frankenbeast CLI — Full E2E Interactive Pipeline

**Date:** 2026-03-06
**Status:** Approved
**Scope:** Make `frankenbeast` a globally-installable CLI that runs end-to-end agent workflows in any project.

## Problem

The `frankenbeast` CLI (`franken-orchestrator/src/cli/run.ts`) is stub-level — only `--dry-run` and `--resume` (display-only) work. Full execution exits with an error. Meanwhile, `plan-approach-c/build-runner.ts` has all the real wiring: observer tracing, RALPH loop execution, git isolation, checkpoint/resume, PR creation. That capability needs to live in the official CLI.

## Design Principles

- **Drop-in tool.** `frankenbeast` works in any project — greenfield or existing. No project-level dependency required.
- **Global install.** `npm install -g franken-orchestrator` (project-local module access is a future iteration).
- **Convention over configuration.** Project state lives in `.frankenbeast/` at the project root. Sensible defaults everywhere.
- **Human-in-the-middle.** Interactive review loops between each phase. The user approves before the CLI advances.

## CLI Surface

### Default: Interactive Session

```bash
frankenbeast                              # interview -> design -> chunks -> exec
frankenbeast --design-doc plan.md         # design -> chunks -> exec
frankenbeast --plan-dir ./chunks/         # chunks -> exec (or resume)
```

When no subcommand is given, the CLI detects the entry point based on what's provided and chains through all subsequent phases with HITM review loops between each.

### Subcommands: Standalone Phases

```bash
frankenbeast interview                    # interview only -> writes design doc
frankenbeast plan --design-doc plan.md    # design doc -> writes chunk files
frankenbeast run                          # exec from .frankenbeast/ chunks (or --plan-dir)
frankenbeast run --resume                 # resume from checkpoint
```

Each subcommand enters at a fixed phase and exits after that phase completes (including its review loop). Subcommands use the same internal Session logic as the default flow.

### Global Flags

```
--base-dir <path>       Project root (default: cwd)
--base-branch <name>    Git base branch (default: main, prompts if current != main)
--budget <usd>          Budget limit in USD (default: 10)
--provider <name>       claude | codex (default: claude)
--no-pr                 Skip PR creation
--verbose               Debug logs + trace viewer on :4040
--reset                 Clear checkpoint and traces
--help
```

### Project Layout

```
<project-root>/
  .frankenbeast/
    config.json                    # optional project-level config overrides
    plans/
      design.md                    # generated by interview
      01_<name>.md, 02_<name>.md   # generated from design doc
    .build/
      .checkpoint                  # execution state for resume
      build-traces.db              # observer traces (SQLite)
      build.log                    # session log
```

## Session Pipeline & State Machine

The `Session` class manages the interactive flow. It determines the entry point based on what's provided, then chains through phases with HITM review loops.

```
                detect
                entry
                point
                  |
        +---------+---------+
        |         |         |
   no files   design doc   chunks
        |         |         |
        v         v         v
   INTERVIEW    PLAN      EXECUTE
        |         |
        v         |
   write design   |
   show paths     |
   review loop    |
        |         |
        v         v
      PLAN -------+
        |
        v
   write chunks
   show paths
   review loop
        |
        v
      EXECUTE
        |
        v
   BeastLoop.run()
   display summary
   PR creation (unless --no-pr)
```

### Review Loop

Each HITM boundary uses the same pattern via the existing `InterviewIO` interface:

```typescript
interface InterviewIO {
  ask(question: string): Promise<string>;
  display(message: string): void;
}
```

```
loop:
  display file paths written
  ask "Would you like to proceed, or is there something you'd like to change?"
  if proceed -> break
  if change -> collect change description -> regenerate -> show paths again
```

### Base Branch Detection

1. Read current branch via `git rev-parse --abbrev-ref HEAD`
2. If `main` -> use silently
3. If not `main` -> prompt: "You're on branch `<name>`. Are you sure you want to target this as your base branch? (y/n)"
   - y -> use it
   - n -> switch to main or let user specify
4. `--base-branch` flag overrides all detection (no prompt)

## Dependency Wiring

The Session constructs all deps that the build-runner currently constructs manually. These are extracted into a reusable factory.

### Observer Stack (from franken-observer)

- `TokenCounter` (zero-arg)
- `CostCalculator(DEFAULT_PRICING)`
- `CircuitBreaker({ limitUsd: budget })`
- `LoopDetector({ windowSize: 3, repeatThreshold: 3 })`
- `SQLiteAdapter(dbFile)`
- `TraceContext.createTrace('FRANKENBEAST Build')`
- Optional: `TraceServer({ adapter: sqlite, port: 4040 })` when `--verbose`

Assembled into an `IObserverModule` implementation with `startTrace`, `startSpan`, `getTokenSpend`.

### CLI Execution Stack

- `RalphLoop()` (zero-arg)
- `GitBranchIsolator({ baseBranch, branchPrefix: 'feat/', autoCommit: true, workingDir })`
- `CliSkillExecutor(ralph, gitIso, observerDeps, verifyCommand, logger)`

### Passthrough Stubs

Modules not used in CLI mode get passthrough stubs (same as build-runner):
- `IFirewallModule` — passes input through
- `IMemoryModule` — no-ops
- `IPlannerModule` — throws (graphBuilder is used instead)
- `ICritiqueModule` — auto-passes
- `IGovernorModule` — auto-approves
- `IHeartbeatModule` — returns empty
- `ISkillsModule` — discovers chunk files as `cli:` skills

### Other Deps

- `FileCheckpointStore(.frankenbeast/.build/.checkpoint)`
- `PrCreator({ targetBranch: 'main', disabled: false, remote: 'origin' })` (unless `--no-pr`)
- `BeastLogger({ verbose, captureForFile: true })`
- `GracefulShutdown` — SIGINT handler: finalize traces, save checkpoint

## File Outputs

### Design Doc Persistence

After `InterviewLoop` generates a design, write to `.frankenbeast/plans/design.md` before the review loop. On regeneration (user requests changes), overwrite the same file.

### Chunk File Persistence

After `LlmGraphBuilder` decomposes the design, write numbered `.md` files to `.frankenbeast/plans/`. Each file contains the chunk's objective, files to modify, success criteria, and verification command. On regeneration, clear and rewrite all chunk files.

## Architecture

### File Structure (new/modified files in franken-orchestrator)

```

  run.ts               # REWRITE: thin entry point -> Session
  config-loader.ts     # KEEP: already handles env/file/defaults
  session.ts           # NEW: interactive pipeline state machine
  dep-factory.ts       # NEW: constructs BeastLoopDeps from CLI args
  review-loop.ts       # NEW: generic HITM review loop
  base-branch.ts       # NEW: branch detection + prompt logic
  project-root.ts      # NEW: detect root, scaffold .frankenbeast/

```

### How Subcommands Map to Session

| Invocation | Entry Phase | Exit After |
|------------|-------------|------------|
| `frankenbeast` (no args) | INTERVIEW | EXECUTE complete |
| `frankenbeast --design-doc X` | PLAN | EXECUTE complete |
| `frankenbeast --plan-dir X` | EXECUTE | EXECUTE complete |
| `frankenbeast interview` | INTERVIEW | INTERVIEW review loop |
| `frankenbeast plan` | PLAN | PLAN review loop |
| `frankenbeast run` | EXECUTE | EXECUTE complete |

## Future Expansion Points (documented, not implemented)

### 1. Additional CLI Providers

The `RalphLoop` provider list is currently `'claude' | 'codex'`. To add arbitrary CLI-based coding agents (aider, cursor-cli, etc.):
- Introduce a `ProviderRegistry` that maps provider names to spawn configs (command, args, rate-limit detection patterns, promise-tag format)
- `RalphLoopConfig.provider` becomes `string` instead of a union
- Each provider registers its CLI command and stderr patterns for rate-limit detection
- The `--provider` flag accepts any registered provider name

### 2. Project-Local Module Access

`npm install franken-orchestrator` as a project dependency for programmatic access to individual modules:
- Import firewall, critique, governor, etc. directly in user code
- Use as middleware in existing agent frameworks
- Separate from global CLI installation — the CLI and the library share the same package but serve different use cases
