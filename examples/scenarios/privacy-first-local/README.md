# Privacy-First Local Deployment

A 100% self-hosted scenario that keeps all data on your infrastructure. Ollama handles inference, regex-based PII masking strips sensitive data before it reaches the model, and ChromaDB provides semantic memory -- all without a single cloud API call.

Designed for organizations operating under GDPR, HIPAA, or internal policies that forbid sending production data to third-party services.

## What This Demonstrates

### PII Detection and Masking

Before any text reaches the LLM, a deterministic regex-based PII masker scans for and replaces sensitive data patterns:

| PII Type | Pattern | Replacement |
|----------|---------|-------------|
| Email | `user@domain.com` | `[EMAIL_REDACTED]` |
| SSN | `123-45-6789` | `[SSN_REDACTED]` |
| Credit Card | `4111-1111-1111-1111` | `[CC_REDACTED]` |
| Phone | `555-123-4567` | `[PHONE_REDACTED]` |

The masker is purely local and deterministic -- no ML models, no cloud services, no network calls. It operates at the application layer before the request enters the Firewall's adapter pipeline. Patterns are applied in priority order, and overlapping matches are deduplicated so that a single piece of text is never double-masked.

In the full framework, this maps to the Firewall's PII interceptor, which runs as an inbound guardrail. The interceptor applies the same regex-based approach but is configurable through `guardrails.config.json` with support for custom patterns, allow-lists, and per-field policies.

### Zero-Cost Local Inference

The scenario uses the `OllamaAdapter` from `frankenfirewall` to route all inference through a locally running Ollama instance. This means:

- No API keys required
- No data leaves your network
- Zero cost per request (local GPU/CPU inference)
- The full IAdapter pipeline is exercised: `transformRequest` -> `execute` -> `transformResponse`
- Token counts are tracked through the `UnifiedResponse` schema, with cost always $0.00

### ChromaDB for Local Semantic Memory

ChromaDB runs as a local Docker container, providing vector storage for semantic search and retrieval-augmented generation (RAG) without any cloud dependency. The scenario includes a health check to verify ChromaDB is reachable. In a production deployment, ChromaDB would store embeddings generated by a local embedding model (e.g., `nomic-embed-text` via Ollama).

### Full Audit Trail

Every request produces a complete audit record that includes:

- **Request ID** -- unique identifier for traceability
- **PII items masked** -- count and types of PII detected and redacted
- **Token counts** -- input and output tokens from the local model
- **Cost** -- always $0.00 for local inference
- **Data sent to cloud** -- always NONE

This audit trail satisfies compliance requirements for GDPR Article 30 (records of processing activities) and HIPAA audit controls. Every interaction is logged with enough detail for a compliance officer to verify that no protected data left the organization's infrastructure.

## Prerequisites

- Node.js 18+
- Docker and Docker Compose (for Ollama and ChromaDB containers)
- Alternatively, [Ollama](https://ollama.ai) installed natively

## Infrastructure Setup

Start Ollama and ChromaDB using Docker Compose:

```bash
npm run infra
```

This runs `docker compose up -d`, which starts:

- **Ollama** on port 11434 with GPU passthrough (falls back to CPU if no GPU)
- **ChromaDB** on port 8000 with persistent volume storage

Then pull a model into Ollama:

```bash
docker compose exec ollama ollama pull llama3.2
```

## Configuration

All configuration is through environment variables. No config files, no cloud credentials.

| Variable | Default | Description |
|----------|---------|-------------|
| `OLLAMA_BASE_URL` | `http://localhost:11434` | Ollama API endpoint |
| `OLLAMA_MODEL` | `llama3.2` | Model to use for inference |
| `CHROMADB_URL` | `http://localhost:8000` | ChromaDB endpoint for health check |

## Run

```bash
npm start
```

## Expected Behavior

1. **ChromaDB health check** -- Reports whether ChromaDB is reachable. The scenario continues either way.
2. **PII detection** -- Scans the test input and identifies 4 PII items (email, phone, SSN, credit card).
3. **PII masking** -- Replaces each PII item with a type-specific placeholder before the text reaches Ollama.
4. **Local inference** -- Sends the masked text to Ollama. No original PII is ever sent to the model.
5. **Audit trail** -- Prints a full audit record showing zero cloud exposure.

## Enterprise Compliance Notes

### GDPR

- **Article 25 (Data Protection by Design)** -- PII is masked at the application boundary before any processing. The LLM never sees raw personal data.
- **Article 30 (Records of Processing)** -- The audit trail provides a complete record of what was processed, what was masked, and that no data left the infrastructure.
- **Article 44 (International Transfers)** -- Not applicable. All processing is local. No data crosses jurisdictional boundaries.

### HIPAA

- **Access Controls (164.312(a))** -- The PII masker acts as a technical safeguard, ensuring PHI is de-identified before reaching the model.
- **Audit Controls (164.312(b))** -- Every request generates a timestamped audit record with PII counts, token usage, and a confirmation that no cloud services were contacted.
- **Transmission Security (164.312(e))** -- All communication is localhost or Docker network. No data traverses external networks.

### Air-Gapped Deployments

This scenario works in fully air-gapped environments. Docker images can be pre-pulled and loaded offline. Ollama models can be imported from files. ChromaDB persists to local volumes. No internet connectivity is required at runtime.
